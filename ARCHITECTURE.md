# RagGov Architecture

RagGov diagnoses RAG failures by turning a run into typed analyzer signals, then selecting a primary failure through an auditable decision policy. The main entry point is `DiagnosisEngine`, which accepts a `RAGRun`, configures native or external-enhanced mode, runs the analyzer pipeline, records fallback and provider-readiness metadata, and returns a structured `Diagnosis`.

The default pipeline starts with retrieval and source-health checks. Retrieval-oriented analyzers build evidence profiles for citation mismatch, stale retrieval, inconsistent chunks, scope violations, temporal source validity, and retrieval diagnosis. These signals describe whether the retrieved context is relevant, current, internally consistent, and sufficient to support the answer. Parser validation can add advisory structure-loss metadata when parser or chunking profiles are absent or degraded.

Grounding and claim-verification stages then inspect whether answer claims are supported by retrieved chunks. Claim grounding, citation faithfulness, and citation probes separate unsupported claims, contradicted claims, post-rationalized citations, and citation mismatches. Sufficiency analyzers add requirement-coverage signals when the retrieved material is too thin to answer the query. These claim-level signals are prioritized before meta-diagnosis because a score or aggregate warning is not itself a diagnosis.

Security stages run alongside the evidence pipeline. Prompt injection, poisoning, retrieval anomaly, and privacy analyzers surface security-relevant findings such as suspicious chunks, prompt override attempts, and unsafe disclosure patterns. The reduced v1 taxonomy treats `PROMPT_INJECTION` as a product-facing primary class and keeps other security findings as traceable sub-signals unless they have enough support to win policy selection.

External packages are optional signal providers. In external-enhanced mode the engine asks the provider registry for readiness, attempts enabled evaluators such as RAGAS, DeepEval, RefChecker, and RAGChecker, and stores missing providers, adapter errors, external signals used, degraded-mode flags, and fallback heuristics in run metadata. External output is advisory: it can block a clean verdict when severe and unexplained, but it does not overwrite native diagnosis.

Analyzer results become decision candidates only when they emit `fail` or `warn` with a failure type. The decision policy assigns each candidate an evidence tier: blocking deterministic, structured diagnostic, heuristic supporting, external advisory, or unknown. It also carries calibration status, signal metadata, method status, evidence strength, analyzer weight, stage, and a concise reason. Weak uncalibrated heuristics, primary-ineligible analyzers, meta heuristics, and external advisory retrieval signals are retained in the trace but suppressed from primary selection when stronger direct evidence exists.

Primary failure selection is deterministic and traceable. The policy filters candidates, prefers fail over warn, ranks by evidence tier, specificity, analyzer weight, calibration status, failure priority, and original order, then applies named exception rules for cases such as citation root causes, stale source lifecycle evidence, grounded-clean overrides, and downstream symptom suppression. The resulting `decision_trace` records the selected analyzer, selected tier, alternatives considered, suppressed candidates, and warnings.

The product-facing v1 taxonomy reduces the many internal failure types to six demoable classes: `CLEAN`, `PROMPT_INJECTION`, `STALE_RETRIEVAL`, `INSUFFICIENT_CONTEXT`, `UNSUPPORTED_CLAIM`, and `CONTRADICTED_CLAIM`. Internal analyzers may still emit finer-grained labels for traceability, but the architecture is designed so every heuristic, external-provider failure, fallback, confidence claim, and suppression decision remains visible rather than silently changing the final verdict.
